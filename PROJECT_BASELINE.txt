================================================================================
TapTapTidy — Project baseline & implementation outline
================================================================================
Last updated: 2026-05-15

Purpose
-------
Chore-tracking application (household tasks, recurring chores, bills/errands, etc.).
This document captures agreed architecture and a phased plan. Do not implement
everything on `main` in one go — use feature branches and small PRs per ticket
below.

--------------------------------------------------------------------------------
Product direction
--------------------------------------------------------------------------------
- Core: track chores and completions (check-off / "tap" workflow).
- Scope: beyond cleaning only — errands, bills, recurring tasks.
- Users: multi-user households (plan for this via Supabase Auth + RLS).
- Platforms:
  - Now: web (Netlify).
  - Later: Android / iOS via Capacitor wrapping the same static SPA build.

--------------------------------------------------------------------------------
Tech stack (decided)
--------------------------------------------------------------------------------
Frontend     SvelteKit (TypeScript)
Mode         SPA — ssr: false (app-like, not marketing/SEO-first)
Adapter      @sveltejs/adapter-static (fallback: index.html for client routing)
Hosting      Netlify (static deploy from SvelteKit build)
Backend      Supabase (Postgres + Auth + client APIs)
Auth         Supabase Auth (email/password or magic link to start; OAuth later)
Security     Row Level Security (RLS) on all user/household data
Styles       SCSS — sass package, lang="scss", @use partials (no Tailwind)
Mobile later Capacitor + same static build (not a separate native rewrite)

Explicitly NOT in v1 baseline
-----------------------------
- Tailwind / UnoCSS
- Full SSR / adapter-netlify server functions (unless a public landing page needs it later)
- svelte-preprocess (unless a future need arises)
- PostCSS (optional later — Autoprefixer only if required)
- Capacitor / store builds (phase 2)

--------------------------------------------------------------------------------
Architecture sketch
--------------------------------------------------------------------------------

  Browser / Capacitor WebView
           |
           v
  SvelteKit SPA (static assets)
           |
           v
  Supabase (Auth + Postgres + RLS)

Web:     Netlify serves static/ + env vars for PUBLIC_* keys
Mobile:  Capacitor loads the same dist/ (add in a later phase)

--------------------------------------------------------------------------------
SvelteKit configuration notes
--------------------------------------------------------------------------------
- ssr: false (globally or per-route tree for the authenticated app)
- adapter-static with fallback: 'index.html' for SPA routing on Netlify and Capacitor
- prerender: consider prerender = true only for truly static public pages (optional later)
- Environment variables (public, safe in client):
    PUBLIC_SUPABASE_URL
    PUBLIC_SUPABASE_ANON_KEY
- Never commit .env; document keys in .env.example only

--------------------------------------------------------------------------------
SCSS pipeline
--------------------------------------------------------------------------------
- Dev dependency: sass
- Component styles: <style lang="scss"> in .svelte files (scoped by default)
- Shared partials: @use '../styles/variables' as *; (prefer @use over @import)
- Global once: import src/styles/global.scss from +layout.svelte
- Tokens: define spacing, colors, radii; expose theme via CSS custom properties in :root
- Mobile-ready conventions (apply in global/layout, not only at Capacitor time):
    - min ~44px touch targets
    - env(safe-area-inset-*) on main shell
    - 100dvh for full-height layouts where needed

--------------------------------------------------------------------------------
Supabase — data & auth (high level)
--------------------------------------------------------------------------------
Auth
  - Sign up, sign in, sign out, session persistence in browser
  - profiles row linked to auth.users (on signup trigger or app insert)
  - OAuth / deep links: defer until Capacitor phase

Suggested tables (names flexible; refine in schema ticket)
  - profiles          (id = auth user, display name, ...)
  - households        (name, ...)
  - household_members (household_id, user_id, role)
  - chores            (household_id, title, recurrence, assignee, ...)
  - chore_completions (chore_id, completed_at, completed_by)

RLS
  - Users read/write only rows for households they belong to
  - No sensitive operations relying on client-only checks

--------------------------------------------------------------------------------
Netlify deployment
--------------------------------------------------------------------------------
- Build: npm run build (SvelteKit static output)
- Publish directory: per adapter-static output (typically build/ or dist/ — confirm in svelte.config)
- Node version: pin in .nvmrc or netlify.toml
- Env vars in Netlify UI: PUBLIC_SUPABASE_URL, PUBLIC_SUPABASE_ANON_KEY
- Preview deploys per PR (recommended)

--------------------------------------------------------------------------------
Git workflow
--------------------------------------------------------------------------------
- Default branch: main (protected if possible)
- One ticket ≈ one feature branch ≈ one focused PR
- Branch naming: feature/<ticket-short-name> or chore/<name>
- Merge via PR after review; no large "init everything" commit on main
